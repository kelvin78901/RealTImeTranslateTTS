# RealTimeTranslateTTS UI 评估与优化方案（基于当前版本 + iteration-roadmap）

> 文档目标：针对“当前 UI 简陋、信息结构不清”的问题，给出可执行的升级方案。  
> 基线版本：当前主干（已包含 SWR、上下文、术语库、历史回填）。  
> 对齐路线：`docs/iteration-roadmap.md`（Iter-1 ~ Iter-7）。

## 1. 当前 UI 评估（问题拆解）

### 1.1 优点（可保留）

1. 功能覆盖广：ASR/翻译/TTS/术语库/延迟模式均已可配置。
2. 工程链路完整：从识别到翻译到播报有闭环，且支持历史记录。
3. 技术能力已超出“演示级”：SWR、background、domainHint 等高级能力已存在。

### 1.2 核心问题（为什么会“简陋”）

1. **信息密度过高、分组不稳定**  
   配置项多但层级浅，用户在主页面同时看到太多“系统级参数”，缺少任务导向结构。

2. **主流程与高级配置混在一起**  
   “开始翻译”主任务路径与模型/引擎/术语源管理等高级项没有明确隔离。

3. **状态可视化不足**  
   SWR、术语命中、回退原因、当前模式收益（速度/质量）缺少直观反馈。

4. **跨场景一致性弱**  
   主界面与悬浮窗在信息呈现、操作心智上不够统一。

5. **为后续迭代预留不足**  
   Iter-1（流式 ASR）、Iter-4（语种识别）、Iter-5（说话人）、Iter-6（AI 助手）将显著增加 UI 复杂度，当前布局难以承载。

---

## 2. 设计目标（本次方案的“成败标准”）

### 2.1 产品目标

1. 新用户 30 秒内完成“录音 → 翻译 → 播报”。
2. 高级能力（SWR/术语库/background）可用但不打扰主流程。
3. 为 Iter-1~Iter-7 预留稳定扩展位，避免每次迭代引发 UI 重构。

### 2.2 体验目标

1. 首屏只承载“当前任务必须信息”。
2. 状态统一：同一语义在主页面与悬浮窗表现一致。
3. 可解释：关键结果（优译、回退、术语命中）可追踪。

## 3. 新版信息架构（IA）

### 3.1 三层结构

1. **运行层（Run）**：录音、翻译流、当前模式、关键状态。  
2. **增强层（Enhance）**：SWR、背景信息、领域词库、术语源。  
3. **系统层（System）**：引擎选择、模型下载、API 配置、调试面板。

### 3.2 页面结构建议

#### A. 主工作台（默认页）
- 顶栏：会话标题 + 模式（实时/平衡/质量）+ 状态点（在线/离线）。
- 中间：实时转写与翻译流（按段落卡片）。
- 底部：主操作区（录音、播报、暂停、快速模式切换）。

#### B. 增强配置抽屉（底部弹层）
- 背景信息输入（background）
- 领域选择（auto/general/meeting/...）
- 术语管理入口（下载源、上传词库）

#### C. 系统设置页（二级页）
- ASR/翻译/TTS 引擎
- 模型管理与下载
- API 检测、设备监控、实验性开关

## 4. 核心交互改造（当前版本可落地）

### 4.1 翻译卡片（Conversation Card）

每条结果卡片采用统一结构：

- 上行：英文原句（可折叠）
- 下行：当前译文（快译或优译）
- 状态徽标：`快译` / `已优化` / `回退` / `错误`
- 右上角 `ⓘ`：打开元信息（route/domain/glossary/background）

### 4.2 SWR 可视化规则

1. 快译到达：显示 `快译` 标签。
2. 优译覆盖：标签切为 `已优化`，并显示轻微高亮 1.5s。
3. 质量路径失败：保留快译，显示 `回退` 提示（不打断播报）。

### 4.3 控件分层

- 主界面仅保留：录音、停止、模式切换、是否自动播报。
- background/domain/术语源移动至“增强配置抽屉”。
- API Key、模型下载等统一收敛到系统设置页。

### 4.4 悬浮窗对齐

- 悬浮窗保留极简：录音按钮 + 最近一句 + 当前状态点。
- 点击“展开”进入与主页面一致的简化卡片视图。
- 同样显示 `已优化` 标记，避免主/浮窗语义不一致。

## 5. 与迭代路线的映射（关键）

### Iter-1：流式 ASR 替换 Vosk
- UI 增加“流式识别状态条”：`Listening / Decoding / Endpoint`。
- 部分结果（partial）用浅色实时滚动；final 才进翻译卡片。

### Iter-2：Kokoro TTS + 多音色
- 新增“语音预览器”组件（5 秒试听）。
- TTS 卡片增加音色标签与语速可视滑条。

### Iter-3：降噪 + 标点恢复
- ASR 结果行增加“增强标识”图标（noise/punc on）。
- 可在调试页查看“增强前后对比”。

### Iter-4：语种自动识别 + 双向互译
- 顶栏语言对从固定 `EN→ZH` 升级为自动/手动切换器。
- 每段卡片显示 `detected_lang`，支持反向翻译快捷按钮。

### Iter-5：说话人识别
- 卡片左侧加入说话人胶囊（A/B/C）。
- 会议模式下按说话人分段聚合。

### Iter-6：AI 助手对话
- 增加“助手侧栏”，可对当前会话执行：总结、追问、草拟回复。
- 回答引用原文片段，点击可定位到会话节点。

### Iter-7：会议专用功能
- 新增“会议工作台”页：议题、行动项、待确认、纪要导出。
- 录音结束后自动生成会议摘要草稿。

## 6. 视觉系统升级建议（不改品牌前提）

### 6.1 视觉风格

1. 卡片化 + 轻层次阴影（弱拟物，突出可读性）。
2. 主色只用于“当前动作与状态”，避免大面积色块。
3. 字体层级固定：标题/正文/辅助信息三档。

### 6.2 关键样式规范

- 圆角：12dp（卡片）/ 20dp（胶囊按钮）
- 间距：8dp 基线网格
- 状态色：
  - success（已优化）`#2E7D32`
  - warning（回退）`#ED6C02`
  - error（失败）`#D32F2F`
  - info（快译）`#1976D2`

### 6.3 可访问性（A11y）

1. 所有主操作点击区域 ≥ 48dp。
2. 状态不只靠颜色，必须有图标/文案。
3. TalkBack 描述补齐（如“已优化译文，点击查看原因”）。

## 8. 验收指标（UI 版本上线门槛）

### 8.1 任务效率

1. 新用户首次完成“录音→翻译→播报”耗时 < 30 秒。
2. 常用设置操作（模式切换/领域切换）点击次数下降 ≥ 30%。

### 8.2 可理解性

1. 80% 用户能说清“快译与已优化”的差异。
2. 90% 用户能找到“术语与背景信息”入口。

### 8.3 稳定性

1. UI 重绘导致掉帧无明显上升（对比当前基线）。
2. 悬浮窗与主界面状态一致率 100%。

---

## 9. 风险与规避

1. **功能过多导致再次拥挤**  
   规避：严守三层结构，禁止系统配置回流主界面。

2. **迭代引入新能力后 UI 再次碎片化**  
   规避：所有新能力先映射到 Run/Enhance/System 三层再实现。

3. **状态文案不统一**  
   规避：建立统一状态词典（快译、已优化、回退、失败）。
